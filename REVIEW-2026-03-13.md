## 【AWS PSA 视角】

**整体判断**：方案选型正确，S3 Vectors 确实是这个场景的最佳选择——轻量、无基础设施、成本低。代码结构清晰，16 个 CRUD 脚本 + 路由子系统分层合理。但有几个技术细节需要修正。

### ⚠️ P0 — `embed.py` 全局单例 Bug

```python
_embed_client = None

def _get_client(region=None, profile=None):
    global _embed_client
    if _embed_client is None:
        # ...创建 client
    return _embed_client
```

问题：`_get_client` 接受 `region` 和 `profile` 参数，但一旦首次初始化后，后续不同参数的调用都会复用第一个 client。在 `benchmark.py` 里这可能不会暴露（因为 region 不变），但如果有人在同一进程里先调 `us-east-1` 再调 `ap-northeast-1` 的 embed，就会静默地用错 region。

→ 修复：要么缓存 key 加上 `(region, profile)` 元组，要么每次都创建新 client（Bedrock 连接池开销很小）。

### ⚠️ P1 — `text[:8000]` 截断是字符级，不是 token 级

Titan v2 限制是 8,192 **tokens**，不是字符。中文密度约 1.5 char/token，8000 个中文字符 ≈ 5,300 tokens，不会超限。但纯英文 8000 chars ≈ 2,000 tokens，远没到上限反而浪费了。这个截断逻辑虽然不会导致报错（短了不会超限），但不够精确。

→ 建议：对于 Skill 路由场景文本都很短（描述通常 < 500 字），这个问题实际影响很小。加个注释说明即可，不用引入 tiktoken。

### P1 — `build_skill_index.py` 的异常判断脆弱

```python
if "NotFound" in str(e) or "NotFound" in err_code or "NotFoundException" in type(e).__name__:
```

用字符串匹配异常类型不可靠。boto3 的 `client.exceptions.NotFoundException` 是结构化的，应该用：
```python
except client.exceptions.NotFoundException:
    # 创建
except ClientError as e:
    if e.response['Error']['Code'] in ('NotFoundException', 'NoSuchBucket'):
        # 创建
    else:
        raise
```

### P2 — 跨 Region 调用链

Embedding 走 `us-east-1`，S3 Vectors 走 `ap-northeast-1`。每次路由查询都要跨太平洋一个来回（~150ms RTT）。从我们内部经验来看，Titan Embeddings v2 在 ap-northeast-1 也是 GA 的，可以考虑同 Region 部署。

### 做得好的地方

- `common.py` 统一 JSON 输出 + 错误处理，所有 16 个脚本风格一致，OpenClaw agent 解析起来很方便
- `embed.py` 内存缓存 + 指数退避重试，考虑了限流场景
- `build_skill_index.py` 自动创建 bucket/index，首次使用零配置
- `benchmark.py` 三种路由模式（Bedrock/TF-IDF/S3）覆盖不同验证场景
- `extract_queries.py` 从真实 session log 提取查询，比手工构造测试集有说服力

**整体判断**：方向对，Skill 膨胀确实是 LLM agent 平台的真实痛点。但 Blog 里的数据呈现有几个严重问题，如果发出去会损害技术可信度。

### ⚠️ P0 — Blog 展示的命中结果明显有误，与"95%+ 命中率"主张矛盾

| 查询 | Blog 列出的 Top-3 | 问题 |
|------|-------------------|------|
| "用 message 工具发送 Slack 消息" | discord, wacli, humanizer-zh | ❌ Slack skill 完全没命中！这是最基础的关键词匹配 |
| "COS 版为什么能每条消息前实时拦截" | discord, blogwatcher, camsnap | ❌ 与查询完全无关 |
| "可以使用 aws 的 s3 向量功能吗" | aws-knowledge, aws-cloudwatch, aws-neptune | ❓ s3-vector-bucket skill 自己都不在 Top-3？ |

Blog 声称"Top-3 命中率 95%+"，但实际展示的 6 个例子中至少有 2 个是明显错误的。作为架构决策者，如果我看到声称 95% 但例子里自相矛盾，我会对整个方案的可信度打折扣。

→ 修复：要么重新跑 benchmark 用真实 S3 Vectors 模式（`--use-s3`），确认数据准确后再贴；要么诚实标注"部分查询命中不理想，语义路由在工具类查询上存在局限性"。**不要用不准确的数据去支撑一个过高的结论。**

### ⚠️ P0 — 数字不一致，标题/正文/README 三处不同

- Blog 标题："91%"
- Blog 正文："90.8%"
- README："92%"
- SKILL.md："92%"

这到底是 91% 还是 92%？看 benchmark 代码，`count_tokens()` 用的是 `zh/1.5 + en/4` 的粗略估算，连 tiktoken 都没用。所以这些数字本身就是近似值，精确到 90.8% 没有意义。

→ 建议：全文统一为"约 91%"或"~90%"，不要给出假精度的小数点。在 methodology 里注明 token 计数方式是近似估算。

### P1 — Hook 的实际效果远不如 Blog 宣传

Blog 给人的印象是"每轮对话节省 91%"。但实际上：

- Hook 只在 `agent:bootstrap` 时触发一次
- bootstrap 时用的上下文是"最近 2 天 memory 文件的最后 500 字符"——不是当前用户消息
- 一个 session 内后续所有消息，用的都是 bootstrap 时选的那 5 个 Skill
- `message:received` hook 还不存在（OpenClaw 文档标注为 "Future Events"）

这意味着**真实场景下，一个包含多种话题的 session 只有第一轮可能命中**。如果用户先问天气、再问 EKS、再问 GitHub，bootstrap 时选的 5 个 Skill 最多覆盖其中一个方向。

→ Blog 应该明确区分"理论上限"（逐消息路由，91%）和"当前实际"（仅 bootstrap，效果有限）。当前架构离"91% 节省"的落地还有一段距离。

### P1 — Hook 的 shellEscape + CLI 传参有安全和长度风险

```typescript
"--query", shellEscape(context.slice(0, 500)),
```

把 memory 内容拼进 shell 命令行参数：
1. Linux `ARG_MAX` 通常 2MB，500 字符不会超限，但如果未来调大就可能 crash
2. `shellEscape` 只做单引号包裹，如果内容包含 null 字节或极端字符，仍可能出问题
3. 更稳健的方式是通过 stdin pipe 传入

### P2 — Blog 缺少成本分析

Blog 花大量篇幅讲 Token 节省，但没算 Skill 路由本身的成本：
- 每次路由 = 1 次 Bedrock Titan Embeddings 调用 + 1 次 S3 Vectors 查询
- Titan Embeddings v2 定价：$0.00002/1K input tokens
- S3 Vectors 查询定价：需要查具体 pricing page

如果每条消息都路由，增加的 Bedrock 调用成本 vs 节省的 LLM input token 成本，净收益是多少？Blog 应该给出 break-even 分析。
